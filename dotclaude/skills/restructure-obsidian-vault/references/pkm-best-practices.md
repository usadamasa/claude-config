# PKM ベストプラクティス参考資料

## PARA方式の概要

Tiago Forte が提唱する情報整理フレームワーク:

| カテゴリ | 説明 | 例 |
|---|---|---|
| **P**rojects | 期限のある進行中の取り組み | 新機能開発、移行プロジェクト |
| **A**reas | 継続的に管理する責任領域 | チーム運営、コンポーネント保守 |
| **R**esources | 将来参照する可能性のある情報 | 技術調査メモ、Runbook |
| **A**rchives | 完了・非アクティブな項目 | 終了プロジェクト |

### PARA方式との対応例

- Projects → 特定プロジェクトフォルダ (例: `Data Fabric開発/`)
- Areas → `Component/`, `DevOps/`
- Resources → `Tech/`, `運用調査Runbook/`
- Archives → `99_Archives/`

## フォルダ命名規則のベストプラクティス

### 数字プレフィックス

- `00_`, `01_`, `02_`, `99_` のようなプレフィックスでソート順を制御
- 利点: ファイラーやサイドバーでの表示順が安定する
- 注意: すべてのフォルダにプレフィックスを付ける必要はない｡ワークフロー上の優先度が高いもの(Daily, Weekly, Inbox)に限定するのが実用的

### 言語の統一

- 日本語フォルダ名と英語フォルダ名が混在する場合がある
- 統一ルールを設けるかは運用次第だが、混在自体は問題にならない
- Obsidianのwiki linkはフォルダパスを含まない `[[note名]]` 形式なので、フォルダ名変更によるリンク切れリスクは低い

## Obsidian特有の考慮事項

### Wiki Link形式とリンク安全性

- `[[note name]]` 形式のリンクはファイル名のみで解決される
- フォルダパスを含まないため、ディレクトリ構造の変更でリンクが壊れにくい
- Obsidian設定で「リンク自動更新」が有効であれば、ファイル移動時にリンクが自動更新される

### ディレクトリ構造変更時のチェックリスト

1. Dataviewクエリでフォルダパスを直接指定しているノートがないか確認
2. テンプレート内のフォルダパス参照を確認
3. Obsidian設定 (newFileFolderPath, attachmentFolderPath) の整合性を確認
4. CLAUDE.mdのディレクトリ構成テーブルを更新
